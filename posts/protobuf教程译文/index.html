定义消息类型（Defining A Message Type）

一个简单的例子
syntax = "proto3";							// 指定 protobuf 的版本，若不指定，则默认为 proto2，且必须是第一个非空非注释行

message SearchRequest {
  string query = 1;
  int32 page_number = 2;
  int32 result_per_page = 3;
}

定义名为 SearchRequest 的 message 类型：

在示例中，都是 scalar type，包含两个整数（result_per_page, page_number）和一个字符串类型（query）。字段也可以指定为组合类型，比如：枚举和其他消息类型。

字段编号分配的规则

  • 每个字段都有一个唯一的编号（用于以消息二进制格式标志字段，在确认使用消息类型后不应更改）
  • 1 - 15 编号，只需一个字节进行编码，而 16 - 2047 的字段编号采用两个字节。因此，常用的消息元素应分配 1- 15,同时需要为将来可能添加的频繁出现的元素预留空间
  • 编号分配的最小值为1，最大为2^29 - 1（536,870,911）。禁用：19000 - 19999，因为其已被用于 Protocol Buffers 的实现。
指定字段的规则

消息字段可以有两种选择（前提：都要确保格式良好）：

  • singular：0 <= 字段 <= 1，为 proto3 的默认字段规则
  • repeated：字段 >= 0，重复值的顺序将被保留

在 proto3 中，scalar 数值类型的重复字段默认使用 packed 编码

如：

message Test4 {
  repeated int32 d = 4 [packed=true];
}
添加更多消息类型
在 .proto 文件中，可以定义多个消息类型。如：

message SearchRequest {
  string query = 1;
  int32 page_number = 2;
  int32 result_per_page = 3;
}

message SearchResponse {
 ...
}
注释规则

给 .proto 中添加注释时，多行使用 / * … * / 语法，单行使用 // 语法。如：

/* SearchRequest represents a search query, with pagination options to
 * indicate which results to include in the response. */

message SearchRequest {
  string query = 1;
  int32 page_number = 2;  // Which page number do we want?
  int32 result_per_page = 3;  // Number of results to return per page.
}
预留字段（Reserved Fields）
若一个消息类型作废了，会想当然的将其注释掉或者删除掉该字段，当其他开发者对该类型的字段重新编号（或更新编号时）。可能会导致严重的问题。因此不能直接保留干掉！通用解法是指定保留已删除字段的字段编号（/ 或 名称，这有可能导致产生 JSON 序列化的问题），在此基础上若有开发者使用该字段的标识符，则 protobuf 编译器会发出警告。示例：

message Foo {
  reserved 2, 15, 9 to 11;
  reserved "foo", "bar";
}

注意：不能在同一个保留语句中混合字段名和字段编号。

.proto 文件生成了什么？
  1. 当 protocol buffer compiler 编译 .proto 时（.proto 文件中已经定义了工作的消息类型，获取以及设置字段值），然后从输入流解析消息，将消息序列化到输出流
  2. Go 语言中，protocol buffer compiler 会生成 .pb.go 的文件，文件中的每个消息类型都有一个类型

Protocol Buffers API Reference 规定了更多的细节

默认值（Default Values）

  • 字符串（string）：默认值为空字符串

  • 字节（bytes）：默认值为孔子杰

  • 布尔（bools）：默认值为 false

  • 数值类型（numeric）：默认值为 0

  • 枚举类型（enums）：要求第一个枚举类型的值必须为 0

  • message 字段：如果未指定，则其默认会随着语言的变更而不同

    repeated 字段的默认值为空

注意：

  1. 对于 scalar 的消息字段，一般消息被解析，则会产生二义性。即无法判断该字段是显示设置为默认值还是压根没有设置。因此定义消息类型的时候，需要明确这一点。
  2. 若 scalar 消息字段设置为默认值，则该值将不会在序列化。（Also note that if a scalar message field is set to its default, the value will not be serialized on the wire.）（ps: 其实不太理解 on the wire 中指代的过程。有大佬知道的话，还望解惑）

枚举（Enumerations）

message SearchRequest {
  string query = 1;
  int32 page_number = 2;
  int32 result_per_page = 3;
  enum Corpus {
    UNIVERSAL = 0;			// 第一个值必须是 0
    WEB = 1;
    IMAGES = 2;
    LOCAL = 3;
    NEWS = 4;
    PRODUCTS = 5;
    VIDEO = 6;
  }
  Corpus corpus = 4;
}

注意

  1. 必须有一个零值，这样我们就可以将0作为一个数值
  2. 0 值必须是第一个元素，其中第一个枚举总是默认值
枚举类型如何起别名？
  1. 将 allow_alias 设置为 true
  2. 将相同的值分配给不同的枚举常量

除此之外，起别名是，编译器会发生错误消息

message MyMessage1 {
  enum EnumAllowingAlias {
    option allow_alias = true;
    UNKNOWN = 0;
    STARTED = 1;
    RUNNING = 1;
  }
}
message MyMessage2 {
  enum EnumNotAllowingAlias {
    UNKNOWN = 0;
    STARTED = 1;
    // RUNNING = 1;  // Uncommenting this line will cause a compile error inside Google and a warning message outside.
  }
}

注意事项：

  • 枚举数常数必须在32位整数的范围内
  • 不推荐使用：因为使用的是 varint encoded，且负值效率低
  • 枚举数可能受到语言的限制
  • 在反序列化过程中，无法识别的枚举值将被保留在消息中
保留值（Reserved Values）
enum Foo {
  reserved 2, 15, 9 to 11, 40 to max;
  reserved "FOO", "BAR";
}

使用其他消息类型（Using Other Message Types）

可以使用其他消息类型作为字段类型。

例：声明 Result 的 message，然后在 SearchResponse 中指定 Result 类型的字段

message SearchResponse {
  repeated Result results = 1;
}

message Result {
  string url = 1;
  string title = 2;
  repeated string snippets = 3;
}
导入定义

导入定义时有两种情况

第一种

以上例为准，若 Result 和 SearchResponse 在相同的文件中，则直接引用即可

第二种

若在不同的文件中，则需要导入原始 .proto 文件，因此可以在 .proto 文件的顶部添加一个 import 语句

import "myproject/other_protos.proto";

更新定义的少见情形（移动 .proto 文件）

  1. 将 .proto 文件移动到新位置

    直接这么做的话会出错的

  2. 新建一个伪 .proto 文件，作用替代之前 .proto 文件，并在一次更新所有调用站点。

  3. 使用 import public 将其转发到新位置。（导入包含 import public 语句的 proto 的任何人都可以传递地依赖 import public 依赖项）

例如：

// new.proto
// All definitions are moved here

// old.proto
// This is the proto that all clients are importing.
import public "new.proto";
import "other.proto";

// client.proto
import "old.proto";
// You use definitions from old.proto and new.proto, but not other.proto
搜索导入文件规则

  1. protocol compiler 使用 -I/--proto_path 标志在协议编译器命令行指定的一组目录中搜索导入的文件

  2. 若没有指定，将会在调用编译器的目录中查找

    通常将 –proto_path 标志设置为项目的根目录，并对所有导入使用完全限定的名称

使用 proto2消息类型

  1. 可以将 proto2 的消息类型导入 proto3 中使用
  2. 可以将 proto3 的消息类型导入 proto2 中使用

只是可以，使用的时候，还需要注意版本间的差异产生的问题，比如：proto2 enums 不能直接在 proto3 语法中使用（如果导入的 proto2 消息使用它们的话是没有问题的）

嵌套类型（Nested Types）

可以在其他消息类型中定义和使用消息类型，如下例：Result 是在 SearchResponse 中被定义的

message SearchResponse {
  message Result {
    string url = 1;
    string title = 2;
    repeated string snippets = 3;
  }
  repeated Result results = 1;
}

如果想要在其父消息类型之外重用此消息类型，则：_Parent_._Type_:

message SomeOtherMessage {
  SearchResponse.Result result = 1;
}

至此，可以随心所欲的将信息做各种嵌套

message Outer {                  // Level 0
  message MiddleAA {  // Level 1
    message Inner {   // Level 2
      int64 ival = 1;
      bool  booly = 2;
    }
  }
  message MiddleBB {  // Level 1
    message Inner {   // Level 2
      int32 ival = 1;
      bool  booly = 2;
    }
  }
}

更新消息类型（Updating A Message Type）

如果现在的消息不能再满足业务需要，在不破坏任何现有代码的情况下更新消息类型将非常容易。其规则如下：

  1. 不要更改任何现有字段的字段编号
  2. 如果添加新字段，则任何由使用“旧”消息格式的代码序列化的消息仍可以由新生成的代码解析。您应该记住这些元素的默认值，以便新代码可以正确地与旧代码生成的消息交互。类似地，由新代码创建的消息也可以由旧代码解析：旧的二进制文件在解析时只会忽略新字段。有关详细信息，请参阅“未知字段”部分。
  3. 只要字段号不再在更新的消息类型中使用，就可以删除字段。您可以改为重命名该字段，可能添加前缀“obsolited_u”，或者保留字段编号，这样.proto的未来用户就不会意外地重用该编号。
  4. int32、uint32、int64、uint64和bool都是兼容的—这意味着您可以将字段从这些类型之一更改为另一种类型，而不会破坏前后兼容性。如果从不适合于相应类型的线中解析一个数，则将获得与在C++中将该数转换为该类型的相同效果（例如，如果64位数字被读取为一个IT32，则它将被截断到32位）。
  5. sint32和sint64彼此兼容，但与其他整数类型不兼容。
  6. 只要字节是有效的UTF-8，字符串和字节就兼容。
  7. 如果字节包含消息的编码版本，则嵌入消息与字节兼容。
  8. fixed32与sfixed32兼容，fixed64与sfixed64兼容。
  9. 对于string、bytes和message字段，optional与repeated兼容。给定重复字段的序列化数据作为输入，如果该字段是基元类型字段，则希望该字段为可选的客户端将采用最后一个输入值；如果是消息类型字段，则会合并所有输入元素。注意，对于数值类型（包括bool和enum）来说，这通常是不安全的。数字类型的重复字段可以以压缩格式序列化，当需要可选字段时，将无法正确分析该格式。
  10. enum 在 wire format 方面与int32、uint32、int64和uint64兼容（请注意，如果值不合适，它们将被截断）。但是，请注意，当消息被反序列化时，客户端代码可能会对它们进行不同的处理：例如，无法识别的proto3枚举类型将保留在消息中，但是当消息被反序列化时，如何表示这些类型取决于语言。Int字段总是保留它们的值。
  11. 将单个值更改为新值的成员是安全的，并且二进制兼容。 如果确保没有代码一次设置多个字段，那么将多个字段移动到新的字段中可能是安全的。 将任何字段移动到现有的字段中都是不安全的

未知类型（Unknow Fields）

未知字段是格式良好的协议缓冲区序列化数据，表示解析器不识别的字段。 例如，当旧二进制解析由新二进制发送的带有新字段的数据时，这些新字段将成为旧二进制中的未知字段。

最初，proto3消息在解析过程中总是丢弃未知字段，但在3.5版本中，重新引入了未知字段的保存来匹配 proto2行为。 在3.5及以后的版本中，解析期间保留未知字段，并将其包含在序列化输出中。

Any

Any 允许将消息用作嵌入类型，且无需为其做 .proto 定义。

一个 Any 以 bytes 呈现序列化的 message，且以一个 URL 作为全局唯一标识符。

如果想使用 Any 类型，则需引入 google/protobuf/any.proto

import "google/protobuf/any.proto";

message ErrorStatus {
  string message = 1;
  repeated google.protobuf.Any details = 2;
}

Any 类型可替换 proto2 的扩展

Oneof

若有一个包含多个字段的消息，且最多同时设置一个字段，则可以使用 Oneof 的特性强制执行用于节省内存。

其中一个字段类似于常规字段，只不过共享内存中的一个字段中的所有字段都是常规字段，而且最多可以同时设置一个字段。 设置其中的任何成员都会自动清除所有其他成员。 根据所选择的语言，可以使用特殊 case ()或 WhichOneof ()方法检查 oneof 中的哪个值被设置(如果有的话)。

message SampleMessage {
  oneof test_oneof {
    string name = 4;
    SubMessage sub_message = 9;
  }
}
Oneof 的特点
  1. 设置 oneof 字段将自动清除 oneof 的其他所有成员。因此，即使设置了多个字段，则只有最后的字段的值具有效力
SampleMessage message;
message.set_name("name");
CHECK(message.has_name());
message.mutable_sub_message();   // Will clear name field.
CHECK(!message.has_name());
  1. oneof 不能是 repeated 型
  2. 反射的 APIs 作用于 oneof 字段
  3. 如果将 oneof 字段设置为默认值（例如将 int32 oneof 字段设置为0），则将设置该 oneof 字段的 “case”，并在网络上序列化该值。
  4. 如果你使用 c + + ，确保你的代码不会导致内存崩溃。
// 以下代码将崩溃。因为通过调用 set_name ()方法已经删除了 sub _ message
SampleMessage message;
SubMessage* sub_message = message.mutable_sub_message();
message.set_name("name");      // Will delete sub_message
sub_message->set_...            // Crashes here
  1. 同样在 c + + 中，如果您使用 Swap ()两条消息中的一条，每条消息将以另一条消息中的一条结束
SampleMessage msg1;
msg1.set_name("name");
SampleMessage msg2;
msg2.mutable_sub_message();
msg1.swap(&msg2);
CHECK(msg1.has_sub_message());
CHECK(msg2.has_name());

map 类型（Maps）

如果你想创建一个关联映射作为你数据定义的一部分，使用如下的格式：

map<key_type, value_type> map_field = N;
// 如：
map<int64,string> values = 1;

map字段不能同时使用repeated

Packages

定义proto的包名，包名可以避免对message 类型之间的名字冲突，同名的Message可以通过package进行区分。

在没有为特定语言定义option xxx_package的时候，它还可以用来生成特定语言的包名，比如Java package, go package。

package foo.bar;

定义服务（Defining Services）

如果要在RPC（远程过程调用）系统中使用消息类型，可以在.proto文件中定义RPC服务接口，协议缓冲区编译器将以您选择的语言生成服务接口代码和存根。例如，如果要使用接受SearchRequest 并返回 SearchResponse 的方法定义 RPC 服务，可以在 .proto 文件中定义它，如下所示：

service SearchService {
  rpc Search(SearchRequest) returns (SearchResponse);
}

与协议缓冲区一起使用的最直接的RPC系统是gRPC：一个在Google开发的语言和平台无关的开源RPC系统。gRPC与协议缓冲区配合得特别好，允许您使用特殊的协议缓冲区编译器插件直接从.proto文件生成相关的RPC代码。

如果您不想使用gRPC，也可以在自己的RPC实现中使用协议缓冲区。你可以在Proto2语言指南中找到更多的信息。

JSON Mapping

Proto3 支持 JSON 的规范编码，使得系统之间更容易共享数据。

如果 json 编码的数据中缺少某个值，或者该值为 null，那么在解析为协议缓冲区时，该值将被解释为适当的默认值。

如果 json 编码的数据中缺少某个值，或者该值为 null，那么在解析为协议缓冲区时，该值将被解释为适当的默认值。 如果一个字段在协议缓冲区中具有默认值，为了节省空间，默认情况下 json 编码的数据中将省略该字段。

Options

option可以用在proto的scope中，或者message、enum、service的定义中。 可以是Protobuf定义的option，或者自定义的option。

option的定义格式是"option" optionName "=" constant ";",比如:

option java_package = "com.example.foo";

一些Protobuf定义的option:

  • java_package
  • java_multiple_files
  • java_outer_classname
  • optimize_for
  • cc_enable_arenas
  • objc_class_prefix
  • deprecated

示例：

option (gogoproto.testgen_all) = true;
option (gogoproto.populate_all) = true;
option (gogoproto.benchgen_all) = true;
message NidRepPackedNative {
	repeated double Field1 = 1 [(gogoproto.nullable) = false, packed = true];
	repeated float Field2 = 2 [(gogoproto.nullable) = false, packed = true];
	repeated int32 Field3 = 3 [(gogoproto.nullable) = false, packed = true];
}
